# 제품 브리프

## 최종 목표

`Lucky Six`를 시작점으로 삼아, 공개 URL을 가진 작은 웹 MVP를 만들고 분석, 수익화, AI 기능까지 단계적으로 붙여 1인 운영이 가능한 글로벌 미니 프로덕트로 확장한다.

## 현재 제품

`Lucky Six`는 사용자가 버튼 한 번으로 중복 없는 로또 번호 6개와 보너스 번호를 생성하고 최근 기록을 확인할 수 있는 정적 웹앱입니다.

## 핵심 고객

모바일에서 빠르게 번호를 뽑고 기록을 남기고 싶은 캐주얼 로또 사용자입니다. 초기 타깃은 한국어 사용자이고, 이후에는 국가별 복권 규칙을 선택할 수 있는 글로벌 사용자로 확장합니다.

## 고객 가치 문장

복잡한 가입 없이 즉시 번호를 생성하고, 최근 기록을 확인하며, 모바일에서 보기 좋은 형태로 저장하거나 공유할 수 있게 한다.

## 글로벌 수익형 AI 프로덕트 후보

1. `Lucky Six Insight`: 로또 번호 생성 결과에 대해 AI가 재미용 해석, 공유 문구, 주의 문구를 만들어 주는 엔터테인먼트 도구
2. `Quick Form Concierge`: 작은 사업자가 문의 폼, FAQ, 고객 응대 문구를 AI로 빠르게 생성하는 웹 도구
3. `Launch Page Reviewer`: 1인 창업자가 랜딩페이지 문구, SEO, CTA를 AI로 점검받는 마이크로 SaaS

## 5주 산출물 구분

| 구분 | 목표 산출물 |
| --- | --- |
| 웹 서비스 | 공개 배포된 MVP, 반응형 UI, GitHub 저장소 |
| 데이터 | 사용자 행동 분석, 문의/피드백 수집, 이벤트 로그 |
| 수익화 | 광고 또는 후원 링크, 유료 기능 후보, 결제 검토 문서 |
| 앱 확장 | PWA 개선, 모바일 앱 래핑 여부 결정, 글로벌 언어 확장 |

## 1주차 목표

| 범주 | 완료 기준 |
| --- | --- |
| 첫 웹페이지 제작 | HTML, CSS, JavaScript로 동작하는 페이지가 있다 |
| 배포 | 공개 URL에서 접속 가능하다 |
| 외부 서비스 연동 | GitHub와 Vercel을 연결 가능한 구조로 정리한다 |
| 광고 수익화 준비 | 광고/분석/결제 후보를 정리한다 |

## 1인 창업 역할 분담

| 업무 | 직접 수행 | AI에게 맡길 일 |
| --- | --- | --- |
| 기획 | 고객, 기능, 우선순위 결정 | 아이디어 확장, 경쟁 구도 정리 |
| 개발 | 요구사항 판단, 코드 검수, 배포 승인 | 코드 초안, 테스트 케이스, 리팩터링 제안 |
| 디자인 | 최종 톤, 사용성 판단 | 레이아웃 후보, 문구 후보 |
| 마케팅 | 채널 선택, 실험 판단 | 게시글 초안, SEO 키워드 후보 |
| 운영 | 계정, 결제, 개인정보 책임 | 체크리스트, 운영 문서 초안 |

## 1인 창업의 강점과 약점

강점:

- 의사결정이 빠르다.
- MVP 출시 비용이 낮다.
- 고객 반응에 맞춰 방향 전환이 쉽다.

약점:

- 모든 책임과 운영 리스크가 한 사람에게 몰린다.
- 마케팅, 개발, 고객지원의 병목이 생기기 쉽다.
- 결제, 개인정보, 법무 영역은 외부 검토가 필요할 수 있다.

## 실행 전략

빠르게 출시하고, 실제 사용자 행동과 피드백을 기준으로 작게 고친다. 새 기능은 배포 전 로컬 테스트, 배포 후 프로덕션 확인, 체크리스트 반영까지 완료해야 끝난 것으로 본다.

## 수익화 가설

| 단계 | 모델 | 이유 |
| --- | --- | --- |
| 첫 수익화 | 광고 또는 제휴 링크 | 무료 트래픽 기반 서비스에 붙이기 쉽다 |
| 장기 수익화 | 프리미엄 기능 또는 구독 | AI 해석, 국가별 규칙, 기록 저장 같은 고급 기능에 적용 가능 |
| 전환 순간 | 사용자가 결과를 저장, 공유, 반복 생성하려는 순간 | 사용 의도가 명확하다 |
| 무료에서 유료로 넘어가는 조건 | 여러 회차 기록 저장, AI 공유 문구 생성, 국가별 규칙 선택, 광고 제거가 필요해지는 순간 | 반복 사용자의 불편을 줄이는 기능에 과금한다 |
| 신뢰 요소 | 개인정보 최소 수집, 무작위성 설명, 오락용 안내 문구 | 민감한 기대수익 오해를 줄인다 |

## 커뮤니티 활용 결정

오프라인 해커톤과 커뮤니티는 MVP 검증, 초기 피드백, 배포 사례 공유에 활용한다. 멤버십 자료는 기술 구현보다 출시와 수익화 체크리스트 검증에 우선 사용한다.
